# SeekStar 固定参考流程

状态：语义模型、设计稿与实现的共同 golden fixtures

上位约束：[连续多尺度语义契约](../product/SEMANTIC_SCALE_CONTRACT.zh.md) · [望远镜设计体系](./TELESCOPE_DESIGN_SYSTEM.zh.md)

这些场景用于比较不同版本，避免用随机 AI 输出掩盖模型问题。Fixture 表达语义关系与画面要求，不要求当前代码已经支持全部字段。

## 1. 汽车到飞机：连续横移与返回

### 世界骨架

L0 领域视野包含大小不同、彼此接壤的区域：

- 交通系统：连接汽车、铁路、航空与航运；
- 工程：连接车辆结构、材料、控制与制造；
- 能源与环境：连接燃料、电气化、效率与排放；
- 社会与旅行：连接基础设施、出行和商业服务。

L1 主题视野中，`汽车` 位于交通与工程交界的左侧，`飞机` 位于交通与工程交界的右侧；`公共交通`、`铁路`、`航空运营` 等形成相邻区域。数量不要求与 L0 相等。

L2 解释视野采用可变细化：

| 区域 | Explanation 示例 |
| --- | --- |
| 汽车附近 | 动力系统、底盘与悬架、车身结构、驾驶辅助、充电/燃料网络 |
| 共享交界 | 轻量化材料、控制系统、安全认证、制造自动化、生命周期排放 |
| 飞机附近 | 机翼与气动布局、推进系统、机身结构、航电、适航认证 |

`轻量化材料` 同时属于汽车与飞机；`安全认证` 同时属于工程与公共治理。它们是多父对象，不复制成两份标题。

### 操作脚本

1. 在 L1 将汽车移到视野中心。
2. 向下变焦进入 L2，汽车相关 Explanation 在原位置细化出现。
3. 在 L2 向右平移，经过共享交界，逐渐看到飞机 Explanation。
4. 将飞机区域移到中心。
5. 向上变焦回到 L1。

### 必须看到

- L2 横移过程中没有换 tab、打开子盒子或全量重排。
- 交界内容逐渐混入，而不是汽车内容瞬间替换成飞机内容。
- 返回 L1 后从飞机附近浮出，汽车仍在左侧。
- 层间数量可变，至少存在一对多和多对多关系。
- 父区域轮廓在转场早期以低透明度保留。

## 2. FlashAttention：从方向到解释再到证据

### 世界骨架

L0 领域视野：

- AI 系统；
- 高性能计算；
- 模型效率；
- GPU 软件生态。

L1 主题视野：

- 注意力算法；
- 内存与带宽；
- GPU 编程；
- 模型扩展；
- 框架与部署。

L2 解释视野不是词语列表，而是有角色的一句话对象：

| 角色 | 标题 | Orientation summary |
| --- | --- | --- |
| 机制 | IO 瓶颈 | 标准注意力频繁读写高带宽显存，数据移动而非算术成为关键成本。 |
| 方法 | 分块计算 | 把 Q/K/V 分块留在更快的片上存储中，减少往返显存。 |
| 约束 | 在线 Softmax | 分块仍需保持数值稳定并得到与完整 Softmax 一致的结果。 |
| 实现 | Kernel 融合 | 将多个算子合并，减少中间结果写回和 kernel launch 开销。 |
| 比较 | 不是稀疏注意力 | 它主要改变精确注意力的执行方式，而不是简单删除注意力连接。 |
| 实践 | 框架集成 | PyTorch、CUDA 能力、形状和硬件代际决定可用实现路径。 |
| Evidence Route | 原始论文与代码 | 从论文、官方仓库和框架文档验证算法、限制与版本差异。 |

`IO 瓶颈` 同时属于注意力算法和内存带宽；`Kernel 融合` 同时属于 GPU 编程和框架部署。不得在相邻 chunk 重复生成同名卡片。

L3 只出现经过观察的真实来源，例如原始论文、官方仓库与框架文档。来源数量可以为零，不能用 AI 标题填充空位。

### 必须看到

- L0 是领地而非 20 个等大技术名词。
- L1 能说明可探索的不同方向。
- L2 每个中心对象都回答“是什么、为什么重要”。
- L2 的来源航向能追溯到一个或多个 L1 主题。
- L3 来源支持或修正 L2 Explanation，而不是与 L2 一一配对。

## 3. 来源验证：候选不等于现实

### 成功路径

1. AI 为某个 Evidence Route 提出 URL candidate。
2. candidate 只进入来源队列，地图 L3 不出现 tile。
3. Scout 开始观察，队列显示“观察中”。
4. Scout 返回最终 URL、标题、类型、摘要和时间。
5. 同一 candidate 晋升为 source-backed，L3 对应位置出现 Source Tile。

### 失败路径

1. URL 无法加载、内容无效或验证失败。
2. 主画布保持原样，不出现破损 tile。
3. 来源队列显示失败原因的用户级摘要，可执行重试或替换。
4. 技术详情折叠保存，不把 JSON、HTTP 堆栈或 Provider 输出放到地图中央。

### 必须看到

- candidate、observing、verified、failed 四种状态不会混入世界段或持久化状态。
- 一个来源失败不把整个世界标成失败。
- 一个邻段失败不阻止用户浏览已经就绪的区域。
- 重启后 verified tile 可以恢复，failed candidate 仍留在恢复队列。

## 4. 固定画面验收

每轮设计至少提交以下画面，尺寸同时覆盖 1280×720 和 1600×900：

1. 汽车在中心的 L1；
2. 汽车与飞机交界的 L2；
3. 飞机在中心并向上返回后的 L1；
4. FlashAttention 的 L0；
5. FlashAttention 的 L2 聚焦 Explanation；
6. L3 candidate 尚未验证；
7. L3 source-backed tile 已出现；
8. 邻段生成失败但主地图仍可浏览。

每张 L0–L2 画面还必须同时满足：当前层对象尺寸至少出现三个可辨等级；下一层存在低透明度预告但不能命中；聚焦对象显示 `orientation_summary`；汽车→飞机交界与 FlashAttention 多父解释不能被复制成同名对象。切层录屏必须覆盖快速同向滚轮合并、反向滚轮重定向和 reduced-motion。

验收时同时记录：当前 seed、相机 XY、尺度、中心对象 ID、上层 Region、可见对象数、已验证来源数和错误域。截图只有视觉，没有这些地址信息时，不能证明连续性。
