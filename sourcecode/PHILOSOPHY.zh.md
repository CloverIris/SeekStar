# SeekStar 项目哲学

> 英文规范版本见 [`PHILOSOPHY.md`](./PHILOSOPHY.md)。

SeekStar 从一个观察出发：

大多数搜索框和聊天界面，都假设用户**已经知道该问什么**。

这是一种严重的认知局限。空白的搜索框或聊天框，在无声地要求用户先把意图描述精确，系统才能帮忙。但许多真正有价值的发现，发生在精确意图出现之前。

SeekStar 为**问题形成之前**的那一刻而存在。

它面向 **unknown unknowns（未知的未知）**：那些用户还无法去搜索的领域、词语、来源、概念、矛盾、人物、工具与路径——因为用户尚不知道它们的存在。

## 为什么是 SeekStar，而不是 SearchStar

名字不是装饰，而是产品命题。

* **Search（搜索）** 假设目标已经在用户脑中清晰到可以被命名。搜索框在定位之前，先要求精确。
* **Seek（寻）** 假设用户可能只有一个方向、一个碎片、一种「附近似乎有关」的感觉，或一片值得靠近的区域。产品必须帮助用户在问题尚未成形之前，发现**还可以问什么**。

因此 SeekStar 不是「更好的搜索引擎」，也不是「聊天答题机」。它是一种探索仪器，用于空白输入框让人感到无力、不知所措的那一刻。

人类的进步常常始于未知的未知：那些早已存在的领域、来源与关联，只是站在界面前的人还看不见。SeekStar 要让这种「问题之前」的状态变得可导航。

## 1. 搜索不是中心

SeekStar 不把搜索当作线性的请求—响应行为。

搜索只是进入地图的入口之一。

用户可以从这些地方开始：

* 一个模糊的种子词；
* 一个问题；
* 一个网页；
* 一个每日主题；
* 一片圈选区域；
* 一个段落；
* 一个词；
* 一个字符；
* 一条超链接；
* 一条已保存的探索路径。

搜索框不是产品。**可探索的地形**才是产品。

## 2. 界面是认知地图

SeekStar 把信息变成空间场。

距离暗示语义关系。
聚类暗示主题社群。
星座暗示可识别的领域。
雾区暗示未知但可能有价值的方向。
路径保存用户的足迹。
圈选变成 prompt。
导出把探索变成知识资产。

地图不是为了好看而存在。它是为了帮助用户理解：自己在哪里、周围有什么、下一步可能往哪里去。

## 3. 缩放是语义的

在 SeekStar 里，缩放不只是视觉上的放大。

缩放意味着穿越认知层级：

领域 → 主题 → 来源 → 文档 → 章节 → 段落 → 句子 → 短语 → 词语 → 字符 → Unicode → 字典 → 新的 seed。

任意信息粒度，都可以成为新的起点。

## 4. 偶然发现应当结构化

SeekStar 应当创造意外发现的可能性，但不变成随机噪声。

一次好的 SeekStar 时刻是：

「我本来是来看 A 的，但现在我明白了为什么 B 在附近、为什么 C 显得奇怪、以及为什么 D 值得接着探索。」

这不是推荐流逻辑。这是**空间的、可检查的、带来源意识的**结构化偶然。

## 5. AI 是制图师，不是神谕

AI Agent 不应表现得像全知的答题机器。

它的职责是：

* 组织地形；
* 解释用户圈选的区域；
* 识别父级、子级与兄弟概念；
* 揭示相邻的未知；
* 标记不确定性；
* 连接来源；
* 生成可能的下一步问题。

Agent 必须区分有来源支撑的信息与推断。未知区域必须保持可见的不确定性。

SeekStar 会用到大型模型的知识与知识组织能力，但**不把它作为默认的答题界面**。

Agent 的首要价值，是把**未知固定为可探索的结构**：

* 把模糊方向切分为层级与区域；
* 提示相邻可能性，但不假装它们已被证实；
* 明确标记雾区、推断与弱假设；
* 仅在用户圈选区域并主动提问时进行解释；
* 帮助用户带着**更好的问题**离开，而不只是更快的答案。

产品主循环的大部分，应当无需等待模型输出即可完成：平移、缩放、选择、转换 Scout 观察结果、把粒度变成新 seed。AI 不可或缺，但它服务于探索；探索不服务于聊天。

## 6. 存在先于认识

SeekStar 中有价值的东西，并非都能由模型「想出来」。

世界上已经存在一些东西：

* 网页；
* 论文与 PDF；
* 百科条目；
* 文章、文档与公开参考资料。

这些对象在用户意识到其相关性之前就已经存在。SeekStar 必须能够**观察**它们，而不只是**想象**它们。

在这一哲学里，Playwright 扮演 **Scout（探测器）**。它的职责是检索与观察：

* 打开 URL 或计划中的观察目标；
* 读取标题、可见正文、元数据、链接与检索时间；
* 返回带来源信息的结构化观察结果；
* 观察失败时，明确失败而非静默吞掉。

Scout 不决定含义、不排序真理、不自行把观察变成事实。一次 Scout 观察只是一颗**候选星**，直到用户与地图的来源规则将其提升为持久地形。

这就是为什么 SeekStar 需要 AI 与 Scout 同时存在：

* **AI** 帮助组织用户尚不知道如何提问的未知；
* **Scout** 帮助把已经存在的内容，以来源支撑的方式带入地图。

只有模型推断的地图，会变成幻觉星空。
只有抓取来的网页堆叠，会变成缺少方向的 tile 碎片。
SeekStar 需要制图师的结构，也需要探测器与真实存在的接触。

## 7. 望远镜才是产品

SeekStar 向人类第一次用望远镜观察夜空致敬。

那一刻的意义，不是立刻收到一份排序好的答案列表，而是获得一种新的观看方式：移动、放大、对照、注意意外，并带着更好的问题再次回来。

因此望远镜不只是视觉隐喻，更是产品的**交互宪法**。

| 望远镜动作 | SeekStar 语义 |
| --- | --- |
| 放大 | 进入更细的语义深度：领域 → 主题 → 来源 → 文档 → 段落 → 词语 |
| 拉远 | 回到宏观定位，看清当前区域在整片星域中的位置 |
| 水平移动 | 探索邻近空间，在视口边缘发现 frontier 候选星 |
| 盯住一颗星 | 选中、检查、圈选或提升为来源节点 |
| 换一片天区 | 从词语、链接或圈选区域开启新 tab 或新 seed |

产品的主界面，是星图，以及在其上移动的望远镜。

* 在宏观深度，**Star Gallery（宏观镜头画廊 / Macro Lens Gallery）** 帮助用户在领域、主题区、种子场、雾区与待侦察星之间定位。
* 在内容深度，网页与文档以 **tile 场** 呈现：可导航的真实材料平面，而不是聊天式文本流。
* 在细节深度，段落、句子、词语、字符与字典粒度成为可点击入口。任意粒度都可以成为新的 seed。

输入框、Inspector、侧栏托盘与导出面板，都服务于望远镜；它们不能取代望远镜。

## 8. 用户通过看、移、圈、问来探索

SeekStar 不应强迫所有意图都经过打字 prompt。

用户可以：

* 缩放；
* 平移；
* 悬停；
* 套索圈选；
* 画笔标注；
* 把条目拖入侧栏托盘；
* 从词语创建新 tab；
* 对圈选区域提问；
* 把路径导出为 Markdown。

围绕一片区域画下的圈，是语言出现之前的提问。

## 9. 最终产物不是答案

SeekStar 应当帮助用户产出：

* 一个更好的问题；
* 一张领域地图；
* 一条学习路径；
* 一组带来源支撑的笔记；
* 一份 Markdown 简报；
* 一个新的探索 seed；
* 一条发现路径。

目标不是快速终结好奇心。目标是让好奇心变得**可导航**。
