# SeekStar 项目哲学

> 英文版见 [`PHILOSOPHY.md`](./PHILOSOPHY.md)。
> 产品语义以 [`PRODUCT_CONTRACT.zh.md`](./docs/product/PRODUCT_CONTRACT.zh.md) 与 [`SEMANTIC_SCALE_CONTRACT.zh.md`](./docs/product/SEMANTIC_SCALE_CONTRACT.zh.md) 为准；当前实现完成度以 [`CURRENT_BASELINE.zh.md`](./docs/status/CURRENT_BASELINE.zh.md) 为准。

SeekStar 从一个观察出发：大多数搜索框和聊天界面都假设用户已经知道该问什么。

但很多真正有价值的发现，发生在问题能够被准确命名之前。用户也许只有一个模糊方向、一个陌生词、一段材料，或一种“附近可能有什么”的直觉。SeekStar 为这个时刻存在。

它面向 **unknown unknowns（未知的未知）**：那些用户还无法主动搜索的领域、概念、关系、矛盾、人物、工具、来源与路径，因为用户尚不知道它们存在。

## 1. Seek，而不只是 Search

**Search** 假定目标已经清晰到可以被命名；**Seek** 允许用户先有方向，后形成问题。

因此 SeekStar 不是更花哨的搜索结果页，也不是把回答铺在画布上的聊天壳。搜索、提问、导入网页和打开旧轨迹都只是入口。真正的产品是用户可以观察、移动、比较和返回的认知地形。

SeekStar 的目标不是替用户尽快结束好奇心，而是帮助用户发现：

- 我现在位于什么区域；
- 周围还有哪些可能性；
- 它们为什么相邻、重叠或冲突；
- 下一步值得形成什么问题；
- 哪些内容来自推断，哪些内容有真实来源支撑。

## 2. 界面是一张认知地图

SeekStar 把信息组织成空间场：

- 距离提示语义邻近；
- footprint 与区域提示覆盖范围；
- 地标帮助用户保持方向；
- bridge 揭示跨领域联系；
- fog 诚实表达仍未构建或尚不确定的方向；
- trail 保存用户真实走过的路径；
- 来源对象把地图重新连接到现实材料。

地图不是为了制造视觉奇观。它必须帮助用户理解自己在哪里、周围有什么，以及继续移动会看到什么。只有关键词、没有方向说明和关系的节点，即使排列得很漂亮，也仍然只是词云。

## 3. 一张世界，一台望远镜

每个 Seek tab 是一张连续世界。用户平移、聚焦或改变尺度时，不应进入新的盒子、子场景或平行时空。

望远镜是 SeekStar 的交互宪法：

| 望远镜动作 | SeekStar 语义 |
| --- | --- |
| 放大 | 在当前位置进入更细的语义视野 |
| 拉远 | 保持当前位置，恢复更宏观的方向感 |
| 横向移动 | 在同一尺度穿越相邻、重叠和桥接区域 |
| 聚焦 | 检查对象、关系、解释或来源，而不是创建孤立世界 |
| 换一片天区 | 只有明确创建新 Seed 时，才建立新的世界 |

用户可以从“汽车”进入更细尺度，再横向移动到“飞机”的区域；向上收束时，应从当前的“飞机”附近回到上层，而不是被送回最初的“汽车”入口。这种空间记忆比目录树或盒子套盒子更接近人对世界的自然观察。

## 4. 缩放改变语义视野

SeekStar 当前产品尺度固定为四种视野。它们是同一张地图的不同投影，不是四张等量词表：

| 尺度 | 名称 | 回答的问题 |
| --- | --- | --- |
| L0 | 领域视野 | 这片知识世界由哪些大区域构成，我靠近哪里？ |
| L1 | 主题视野 | 当前区域有哪些主题邻域、分支与交叉问题？ |
| L2 | 解释视野 | 它如何工作、由什么组成、有哪些比较、争议、实践路径和证据方向？ |
| L3 | 来源视野 | 哪些经过观察的真实材料支持、反驳或实例化这些解释？ |

这些尺度必须允许一对多、多对一和多对多的 refinement。对象数量由语义密度和真实差异决定，而不是为了填满屏幕而保持 `1:1:1`。

更细的段落、句子、词语等阅读粒度，可以成为未来的内容阅读能力或新 Seed 入口，但不应被冒充为当前世界地图的额外固定层级。

## 5. 偶然发现应当被结构化

SeekStar 应创造偶然发现，但不能制造随机噪声。

一次好的探索体验是：“我本来在看 A，现在理解了为什么 B 在附近、为什么 C 与它冲突，以及为什么 D 值得继续进入。”

这种 serendipity 必须是空间可追踪、关系可解释、来源可检查的。系统可以展示未知和弱假设，但不能用确定语气把它们伪装成事实。

## 6. AI 是 Cartographer，不是神谕

AI Cartographer 的价值，是把尚未成形的问题固定为可探索结构。它负责：

- 提出语义对象、简短方向说明和对象角色；
- 组织邻近、重叠、桥接、对比与 refinement；
- 揭示未知边界和可能的下一步问题；
- 标记推断、不确定性与来源候选；
- 在用户要求时解释当前区域。

Cartographer 不决定像素坐标，不拥有相机，也不能把候选 URL 宣布为真实来源。AI 可以帮助构图，但不能越过现实验证边界。

产品的大部分浏览循环——平移、缩放、选择、返回和查看已缓存内容——都不应等待模型。AI 服务于探索；探索不服务于聊天。

## 7. 存在先于认知，Scout 连接现实

网页、论文、PDF、图片、文档和公开资料，在用户意识到它们相关之前就已经存在。SeekStar 必须能够观察它们，而不只是让模型想象它们。

Scout / DataService 的职责是：

- 打开候选 URL 或本地来源；
- 读取标题、可见内容、元数据、资源类型和观察时间；
- 返回带 provenance 的结构化观察；
- 在观察失败时明确失败，不静默吞掉；
- 只有观察成功后，才把来源提升为 L3 主画布对象。

Scout 不决定地图的主题意义，Cartographer 也不决定来源是否真实。只有模型推断的地图会变成幻觉星空；只有抓取结果的堆叠会变成没有方向的材料仓库。SeekStar 同时需要制图师的结构和探测器与现实的接触。

## 8. 世界池先于画面

用户看到的是已经存在于世界对象池中的内容，而不是相机移动触发的同步 AI 回答。

相机只向后台发出弱信号：当前位置、尺度、方向和接近边缘的程度。生产器据此调整优先级、预测并补充附近地形；Renderer 只消费已就绪对象。缺失、生成中或失败的区域应以真实 fog 和状态边界呈现。

这条原则保护三件事：

- 交互即时，不被网络延迟绑架；
- 世界连续，不因每次点击重新生成而分裂；
- 已探索内容可缓存、恢复和继续生长。

## 9. 用户通过看、移、圈、问来探索

SeekStar 不应强迫所有意图都先变成文字 prompt。用户可以平移、缩放、悬停、选择、圈选、比较、追踪关系、打开来源、询问当前区域，或把任意有价值的对象变成一个明确的新 Seed。

围绕一片区域画下的圈，本身就是语言出现之前的问题。

输入框、右栏、设置、来源审查和导出面板都服务于望远镜。它们不能取代望远镜，也不能形成与活动地图冲突的第二套状态。

## 10. 最终产物不是一个答案

SeekStar 应帮助用户获得：

- 一个更好的问题；
- 一张可继续浏览的领域地图；
- 一条学习或调查路径；
- 一组有来源支撑的材料与笔记；
- 一个可复用的新 Seed；
- 一段能够恢复的发现轨迹。

目标不是尽快终结好奇心，而是让好奇心变得可导航、可检查、可延续。
